# AI Agent 是什么？从感知到行动的智能体

朋友问我最近在做什么，我说"做 AI Agent"。他愣了一下："就是 ChatGPT 套个壳？"

这个问题我听过太多次了。AI Agent 绝不是 LLM 套壳，它是一个全新的软件范式——从"你问它答"变成了"给它目标，它自己想办法完成"。

这篇文章用最直白的方式讲清楚：AI Agent 到底是什么，它和传统程序有什么不同，以及你什么时候真的需要它。

## 一个最简单的例子

先看两段代码。

**传统程序：**

```python
def get_weather(city):
    api_key = "your-key"
    url = f"https://api.weather.com/v1/{city}?key={api_key}"
    response = requests.get(url)
    return response.json()

result = get_weather("北京")
print(result)
```

你写死逻辑，传参，拿结果。每一步都是确定的。

**AI Agent：**

```python
agent = Agent(
    tools=[search_web, read_file, write_code, execute_command],
    llm=ChatOpenAI(model="gpt-4")
)

agent.run("调研一下 LangGraph 的最新版本，然后写一个示例脚本")
```

看到了吗？你没告诉它"先搜索什么、再打开哪个页面、最后怎么写代码"。你只给了它一个**目标**，它自己决定该做什么、用什么工具、遇到错误怎么处理。

这就是 Agent 和传统程序最本质的区别。

## AI Agent 的核心：感知-推理-行动循环

所有 AI Agent，不管多复杂，底层都是同一个循环：

```
循环 {
    感知 → 获取当前状态（用户输入、环境反馈、工具返回）
    推理 → 基于状态决定下一步做什么
    行动 → 执行选定的动作（调用工具、生成回复、修改状态）
}
```

```text
┌─────────┐     ┌─────────┐     ┌─────────┐
│  感知    │ ──→ │  推理    │ ──→ │  行动    │
│ 感知环境 │     │ 决定行动 │     │ 执行动作 │
└─────────┘     └─────────┘     └─────────┘
      ↑                             │
      └─────────────────────────────┘
         行动结果成为新的感知输入
```

一个具体的例子：你让 Agent "帮我写一封邮件回复客户关于延迟发货的投诉"。

| 步骤 | 感知 | 推理 | 行动 |
|------|------|------|------|
| 1 | 收到你的指令 | 需要查找客户信息和订单状态 | **搜索**邮件系统中的客户邮件 |
| 2 | 找到客户邮件，显示客户很生气 | 需要了解真实发货时间 | **查询**订单系统获取物流状态 |
| 3 | 物流显示明天能到 | 需要写一封道歉+解释+承诺的邮件 | **生成**邮件草稿 |
| 4 | 草稿写好了 | 你可能会想修改，先展示给你看 | **输出**草稿并询问是否发送 |

每一步的"行动结果"又变成下一步的"感知输入"。这个循环一直持续到目标完成或达到终止条件。

## Agent 的三个核心能力

拆开来看，一个能真正干活的 Agent 需要三样东西：

### 1. 推理能力（Reasoning）

Agent 之所以不是"if-else 套壳"，是因为它能根据上下文做决策。遇到意外情况时，它能重新规划：

```
用户："帮我分析这份财报"

Agent 的思考过程：
  → 用户给了一份财报文件
  → 我需要先读取文件
  → 如果文件是 PDF，需要用 PDF 解析工具
  → 如果是 Excel，需要用不同的工具
  → 读取完后分析关键指标
  → 最后生成分析报告
```

这种"想一下再做"的能力来自 LLM 的推理能力（CoT、ReAct 等模式），也是 Agent 的核心智力。

### 2. 工具使用（Tool Use）

没有工具的 Agent 只是一个聊天机器人。有了工具，Agent 才能影响真实世界：

- **信息获取**：搜索网页、查数据库、读文件
- **执行操作**：发邮件、创建工单、部署代码
- **计算分析**：运行 Python 脚本、执行 SQL 查询
- **感知环境**：调用 API、读取传感器数据

工具是 Agent 的"手"和"眼睛"。

### 3. 记忆管理（Memory）

Agent 需要在多次交互中记住有用信息：

- **短期记忆**：当前对话上下文（受限于 LLM 上下文窗口）
- **长期记忆**：跨会话的知识沉淀（存储在外部数据库中）
- **工作记忆**：当前任务的关键状态（任务进行中的中间结果）

没有记忆的 Agent 每次对话都像"失忆症患者"。

## AI Agent vs 传统软件

| 维度 | 传统程序 | AI Agent |
|------|---------|----------|
| **控制流** | 开发者预定义 | Agent 动态决策 |
| **输入** | 结构化参数 | 自然语言目标 |
| **错误处理** | 按代码路径处理 | 可自主重试/降级/换方案 |
| **可扩展性** | 需修改代码加功能 | 加工具即可扩展能力 |
| **确定性** | 相同输入=相同输出 | 相同目标可能不同路径 |
| **适用场景** | 流程固定、高可靠性要求 | 流程开放、需要灵活应对 |

不是说 Agent 比传统程序更好——它更适合**开放性问题**，不适合**确定性流程**。比如"计算工资"你用传统程序，但"分析这季度工资数据异常并给出建议"你才用 Agent。

## 什么时候用 Agent？

**适合用 Agent 的场景：**

- 任务步骤不确定，走一步看一步
- 需要动态使用多种工具/API
- 输入是自然语言，输出需要个性化
- 需要处理意外情况和异常

**不适合用 Agent 的场景：**

- 流程完全固定，不需要决策
- 对延迟极度敏感（Agent 需要思考时间）
- 必须完全确定的结果
- 成本敏感（LLM 调用比函数调用贵得多）

## Agent 的分类（简单了解）

市面上各种 Agent 术语让人眼花缭乱，其实按复杂度分就三类：

1. **简单反射 Agent**：基于当前感知做决策，没有内部状态。比如一个"如果检测到 @ 就回复"的自动回复机器人。
2. **基于模型的 Agent**：维护内部状态（记忆），能规划多步行动。大部分生产级 Agent 属于这一类。
3. **学习型 Agent**：能从过往经验中改进自己的策略。比如根据用户反馈调整回复风格的客服 Agent。

你大部分时候接触和开发的是第 2 类。

## 我踩过的坑

**坑 1：把 Agent 当万能药**

刚接触 Agent 时，我觉得什么功能都能用 Agent 实现。结果一个简单的"根据规则路由工单"功能，用 Agent 实现的延迟比硬编码高了 20 倍，而且偶尔还会路由错。**能写死逻辑的地方，就不要用 Agent。**

**坑 2：低估了"推理时间"**

Agent 不是瞬间响应的。一次完整的感知-推理-行动循环需要 1-3 秒。如果你的 Agent 需要 5 个步骤完成任务，用户就要等 5-15 秒。不加流式响应的话，体验极差。

**坑 3：给了太多工具**

一开始我把所有工具都挂到 Agent 上，想着"能力越多越好"。结果 Agent 经常选错工具，或者在决策时因为选项太多而"困惑"。**给 Agent 的工具要精而不是多。**
