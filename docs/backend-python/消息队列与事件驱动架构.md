# 消息队列与事件驱动

Agent 任务可能跑几秒也可能跑几分钟。HTTP 请求-响应模式天然不适合长时间任务：连接超时、中间件切断、用户刷新页面——任何一环断了，正在执行的 Agent 任务就丢了。

消息队列 + 事件驱动架构是解决这个问题的标准方案。

## 什么时候需要消息队列

不是所有 Agent 都需要消息队列。判断标准很简单：

- 你的 Agent 任务会在 5 秒内返回吗？→ 不需要
- 任务需要等待外部系统（人工审批、第三方 API 回调）吗？→ 需要
- 你的用户会频繁刷新页面吗？→ 需要
- 你希望即使 Agent 服务重启，正在进行的任务也不丢吗？→ 需要

## Celery + Redis 实战

Celery 是 Python 最成熟的分布式任务队列，和 FastAPI 配合非常好：

```python
# celery_app.py
from celery import Celery

celery_app = Celery(
    "agent_tasks",
    broker="redis://redis:6379/0",
    backend="redis://redis:6379/1",
    task_serializer="json",
    result_serializer="json",
    accept_content=["json"],
)


# tasks.py
from celery_app import celery_app
from openai import OpenAI

client = OpenAI()


@celery_app.task(bind=True, max_retries=3, soft_time_limit=300)
def run_agent_task(self, session_id: str, user_message: str):
    """Agent 长时间运行任务的异步编排"""
    try:
        # 1. 检索知识库
        context = retrieve_knowledge(user_message)

        # 2. 调用 LLM
        response = client.chat.completions.create(
            model="gpt-4",
            messages=[
                {"role": "system", "content": f"Context: {context}"},
                {"role": "user", "content": user_message},
            ],
            stream=False,
        )

        # 3. 执行工具调用
        result = process_tool_calls(response)

        return {"status": "completed", "result": result}

    except Exception as exc:
        raise self.retry(exc=exc, countdown=60)  # 60 秒后重试
```

### FastAPI 集成

```python
# api.py
from celery.result import AsyncResult


class TaskStatus(str, Enum):
    PENDING = "pending"
    RUNNING = "running"
    COMPLETED = "completed"
    FAILED = "failed"


@app.post("/api/v1/agent/run")
async def start_agent_task(session_id: str, message: str):
    """提交 Agent 任务，立即返回 Task ID"""
    task = run_agent_task.delay(session_id, message)
    return {
        "task_id": task.id,
        "status": "pending",
        "poll_url": f"/api/v1/agent/status/{task.id}",
    }


@app.get("/api/v1/agent/status/{task_id}")
async def get_task_status(task_id: str):
    """轮询任务状态"""
    task = AsyncResult(task_id, app=celery_app)

    if task.state == "PENDING":
        return {"status": TaskStatus.PENDING}
    elif task.state == "RUNNING":
        return {"status": TaskStatus.RUNNING}
    elif task.state == "SUCCESS":
        return {"status": TaskStatus.COMPLETED, "result": task.result}
    elif task.state == "FAILURE":
        return {"status": TaskStatus.FAILED, "error": str(task.info)}

    return {"status": task.state}
```

## 事件驱动架构

消息队列的更深层用法是事件驱动。Agent 的每个生命周期阶段都可以发出事件，不同的处理器消费这些事件：

```python
from dataclasses import dataclass, field
from enum import Enum
import asyncio


class AgentEventType(Enum):
    TASK_STARTED = "task.started"
    LLM_CALLING = "llm.calling"
    TOOL_EXECUTING = "tool.executing"
    TOOL_COMPLETED = "tool.completed"
    TASK_COMPLETED = "task.completed"
    TASK_FAILED = "task.failed"
    HUMAN_INPUT_REQUIRED = "human.input.required"


@dataclass
class AgentEvent:
    type: AgentEventType
    session_id: str
    data: dict = field(default_factory=dict)


class EventBus:
    """轻量级事件总线，生产环境可替换为 RabbitMQ / Kafka"""
    def __init__(self):
        self.handlers: dict[AgentEventType, list] = {}

    def subscribe(self, event_type: AgentEventType):
        """装饰器：注册事件处理器"""
        def decorator(handler):
            if event_type not in self.handlers:
                self.handlers[event_type] = []
            self.handlers[event_type].append(handler)
            return handler
        return decorator

    async def publish(self, event: AgentEvent):
        handlers = self.handlers.get(event.type, [])
        results = []
        for handler in handlers:
            result = await handler(event)
            results.append(result)
        return results


# 订阅者示例：记录日志
event_bus = EventBus()


@event_bus.subscribe(AgentEventType.TASK_STARTED)
async def on_task_started(event: AgentEvent):
    logger.info("task.started", session_id=event.session_id, data=event.data)


@event_bus.subscribe(AgentEventType.TOOL_EXECUTING)
async def on_tool_executing(event: AgentEvent):
    logger.info("tool.executing",
                tool=event.data.get("tool_name"),
                session_id=event.session_id)


# 前端通过 SSE 订阅事件，实现实时推送
@event_bus.subscribe(AgentEventType.TOOL_EXECUTING)
async def notify_frontend(event: AgentEvent):
    await sse_manager.broadcast(event.session_id, event)
```

## RabbitMQ vs Redis vs Kafka

| 特性 | Redis | RabbitMQ | Kafka |
|------|-------|----------|-------|
| 吞吐量 | 中等 | 中等 | 极高（百万级/秒） |
| 消息持久化 | 支持（RDB/AOF） | 支持 | 支持 |
| 消息顺序 | 不保证 | 保证（单队列内） | 保证（分区内） |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 适用场景 | 轻量任务队列、缓存 | 企业级消息路由 | 事件流、日志聚合 |
| Agent 适用度 | ✅ 适合大多数 | ✅ 需要复杂路由时 | ⚠️ 大数据量场景 |

**推荐**：大多数 Agent 项目从 Celery + Redis 起步就够用了。当需要复杂路由（Topic Exchange）或超高吞吐量时再考虑 RabbitMQ 或 Kafka。

## 总结

消息队列让 Agent 系统从"同步请求-响应"升级为"异步事件驱动"。Celery + Redis 是 Python 团队的最优选，事件总线模式则为 Agent 的每个决策阶段提供了可观测和可干预的抓手。选型上遵循"够用原则"：先从 Celery + Redis 开始，遇到瓶颈再升级。
