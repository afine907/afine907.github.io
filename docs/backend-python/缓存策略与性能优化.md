# 缓存策略与性能优化

LLM 调用的延迟通常是几百毫秒到几秒，加上知识库检索、工具执行，一次 Agent 交互可能耗时 5-10 秒。缓存是降低延迟、节省成本最直接的手段——但缓存什么、缓存多久、怎么保证缓存一致性，是三个不同难度的问题。

## 缓存的核心收益

| 场景 | 无缓存 | 有缓存 | 收益 |
|------|--------|--------|------|
| LLM 重复提问（相同 Prompt） | 2-10s | &lt;10ms | 延迟降低 99%+ |
| 知识库检索（相同 Query） | 200-500ms | &lt;5ms | 延迟降低 95%+ |
| 用户会话加载 | 50-100ms (DB) | &lt;5ms (Redis) | 延迟降低 90%+ |
| Embedding 计算（相同文本） | 100-300ms | &lt;1ms | 延迟降低 99%+ |

## 缓存分层架构

```
┌─────────────────────────────────────────┐
│            L1: 内存缓存 (dict/cachetools) │  < 1ms, 进程内
├─────────────────────────────────────────┤
│            L2: Redis 缓存               │  < 5ms, 跨进程共享
├─────────────────────────────────────────┤
│            L3: 数据库持久化              │  50ms+, 最终一致性
└─────────────────────────────────────────┘
```

### L1 内存缓存

适合高频读取、低数据量的场景：

```python
from cachetools import TTLCache, LRUCache

# 基于时间的缓存：TTL 过期
session_cache: TTLCache[str, dict] = TTLCache(maxsize=1000, ttl=300)

# 基于容量的缓存：LRU 淘汰
token_counter_cache: LRUCache[str, int] = LRUCache(maxsize=5000)


async def get_session(session_id: str) -> dict:
    if session_id in session_cache:
        return session_cache[session_id]

    session = await db.get_session(session_id)
    session_cache[session_id] = session
    return session
```

### L2 Redis 缓存

适合跨进程共享、数据量大的场景：

```python
import json
import hashlib
from redis import asyncio as aioredis

redis_client = aioredis.from_url("redis://redis:6379/0")


def make_cache_key(prefix: str, data: str) -> str:
    hash_val = hashlib.sha256(data.encode()).hexdigest()[:16]
    return f"{prefix}:{hash_val}"
```

## LLM 语义缓存

普通的 Key-Value 缓存要求 Prompt 完全一致才能命中。语义缓存根据语义相似度来匹配——意思差不多的提问直接复用之前的回答：

```python
import numpy as np
from openai import OpenAI


class SemanticCache:
    def __init__(self, redis_client, similarity_threshold: float = 0.92):
        self.redis = redis_client
        self.similarity_threshold = similarity_threshold
        self.embedding_client = OpenAI()

    async def get(self, query: str) -> str | None:
        query_embedding = await self._get_embedding(query)

        # 使用 SCAN 分批遍历（避免 KEYS 阻塞）
        cursor = 0
        while True:
            cursor, cached_keys = await self.redis.scan(cursor, match="semantic:*", count=100)
            for key in cached_keys:
                cached_data = json.loads(await self.redis.get(key))
                similarity = self._cosine_similarity(
                    query_embedding, cached_data["embedding"]
                )
                if similarity >= self.similarity_threshold:
                    return cached_data["response"]
            if cursor == 0:
                break

        return None

    async def set(self, query: str, response: str):
        embedding = await self._get_embedding(query)
        key = f"semantic:{hashlib.sha256(query.encode()).hexdigest()[:16]}"
        await self.redis.set(
            key,
            json.dumps({"embedding": embedding, "response": response}),
            ex=3600,  # 1 小时过期
        )

    async def _get_embedding(self, text: str) -> list[float]:
        response = self.embedding_client.embeddings.create(
            model="text-embedding-3-small", input=text
        )
        return response.data[0].embedding

    @staticmethod
    def _cosine_similarity(a: list[float], b: list[float]) -> float:
        a, b = np.array(a), np.array(b)
        return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))
```

语义缓存特别适合 Agent 知识库查询——不同用户问同一个问题，措辞略有不同，但语义是一样的。一次 LLM 调用，全局复用。

## 缓存失效策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| TTL 过期 | 写入后固定时间自动过期 | LLM 响应（1h）、Session（30min） |
| LRU 淘汰 | 淘汰最久未使用的 | 内存缓存容量控制 |
| 主动失效 | 数据变更时主动清除 | 知识库更新后清除相关缓存 |
| 版本号失效 | 带版本号的缓存 Key | 配置变更、模型版本升级 |

```python
class VersionedCache:
    """带版本号的缓存，版本更新时所有缓存自动失效"""

    def __init__(self, redis_client):
        self.redis = redis_client

    async def get_cache_version(self, namespace: str) -> int:
        version = await self.redis.get(f"cache_version:{namespace}")
        return int(version) if version else 1

    async def get(self, namespace: str, key: str):
        version = await self.get_cache_version(namespace)
        return await self.redis.get(f"cache:{namespace}:v{version}:{key}")

    async def set(self, namespace: str, key: str, value, ttl: int = 3600):
        version = await self.get_cache_version(namespace)
        await self.redis.set(
            f"cache:{namespace}:v{version}:{key}", value, ex=ttl
        )

    async def bump_version(self, namespace: str):
        """版本号 +1，所有旧缓存自动失效"""
        await self.redis.incr(f"cache_version:{namespace}")
```

## Agent 场景最佳实践

### 1. Prompt 缓存（System Prompt + Few-shot Examples）

System Prompt 在 Agent 调用中通常是固定的，可以重用：

```python
class PromptCache:
    def __init__(self):
        self.cache: TTLCache[str, str] = TTLCache(maxsize=100, ttl=3600)

    def get_system_prompt(self, agent_type: str, identity: str) -> str:
        key = f"system_prompt:{agent_type}:{identity}"
        if key in self.cache:
            return self.cache[key]

        prompt = self._build_system_prompt(agent_type, identity)
        self.cache[key] = prompt
        return prompt
```

### 2. Token 计数缓存

LLM 的 Token 计数是昂贵的操作，缓存结果：

```python
token_count_cache: TTLCache[str, int] = TTLCache(maxsize=5000, ttl=3600)


def count_tokens(text: str, model: str = "gpt-4") -> int:
    import tiktoken
    key = f"{model}:{hashlib.md5(text.encode()).hexdigest()}"
    if key in token_count_cache:
        return token_count_cache[key]
    encoding = tiktoken.encoding_for_model(model)
    count = len(encoding.encode(text))
    token_count_cache[key] = count
    return count
```

### 3. 会话状态缓存

用 Redis 缓存活跃会话状态，避免每次请求都查数据库：

```python
async def get_session_state(session_id: str) -> dict:
    state = await redis_client.get(f"session_state:{session_id}")
    if state:
        return json.loads(state)

    state = await db.get_session(session_id)
    await redis_client.setex(
        f"session_state:{session_id}", 1800, json.dumps(state)
    )
    return state
```

## 总结

缓存是 Agent 系统性能优化投入产出比最高的手段。L1 内存缓存解决高频热点，L2 Redis 缓存实现跨进程共享，语义缓存是 LLM 场景的特化方案。缓存的难点不在"怎么写"，而在"怎么让缓存失效"。选择合适的失效策略，配合版本号机制，才能构建高性能且正确的缓存系统。
