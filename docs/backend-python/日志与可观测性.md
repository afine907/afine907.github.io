# 日志与可观测性

Agent 系统出问题了你怎么排查？生产环境里 Agent 的决策链路涉及 LLM 调用、工具执行、知识库检索等多个环节，任何一个环节出错都需要快速定位。传统的 `print` 打日志方式在分布式 Agent 系统中完全不够用。

## 可观测性的三大支柱

可观测性（Observability）的三个核心维度：

- **日志（Logs）**：带时间戳的结构化事件记录
- **指标（Metrics）**：可聚合的数值数据（请求数、延迟、错误率）
- **追踪（Traces）**：请求在分布式系统中完整的调用链路

Agent 场景下这三者都不可或缺：日志定位具体错误，指标发现趋势异常，追踪还原 Agent 的完整决策过程。

## 结构化日志

### 为什么需要结构化

```python
# ❌ 不可搜索、不可聚合
print(f"User {user_id} called LLM with prompt {prompt[:50]}...")

# ✅ 结构化日志，可搜索、可过滤
import structlog

logger = structlog.get_logger()
logger.info("llm.call", user_id=user_id, prompt_length=len(prompt), model="gpt-4")
```

`structlog` 比标准库的 `logging` 更适合 Agent 场景：

```python
import structlog

structlog.configure(
    processors=[
        structlog.stdlib.filter_by_level,
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.dev.ConsoleRenderer()  # 开发环境
    ],
    wrapper_class=structlog.stdlib.BoundLogger,
    context_class=dict,
    logger_factory=structlog.stdlib.LoggerFactory(),
    cache_logger_on_first_use=True,
)

logger = structlog.get_logger()
```

### Agent 场景的日志上下文

Agent 的每个请求都有自己的 Request ID，贯穿所有日志：

```python
from contextvars import ContextVar
import uuid

request_id_var: ContextVar[str] = ContextVar("request_id")


@app.middleware("http")
async def add_request_id(request: Request, call_next):
    request_id = str(uuid.uuid4())
    request_id_var.set(request_id)
    response = await call_next(request)
    response.headers["X-Request-ID"] = request_id
    return response


def get_logger():
    return structlog.get_logger().bind(request_id=request_id_var.get())
```

这样每条日志都会带上 Request ID，从用户请求到 LLM 调用再到工具执行，全程可追溯。

## OpenTelemetry 与分布式追踪

Agent 系统的调用链比普通 Web 服务长得多：

```
User Request → Auth → Agent Orchestrator → LLM Call → Tool Execution → Database → Response
```

用 OpenTelemetry 实现端到端追踪：

```python
from opentelemetry import trace
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

tracer = trace.get_tracer(__name__)

# 自动埋点
FastAPIInstrumentor.instrument_app(app)
HTTPXClientInstrumentor().instrument()

# 手动埋点：Agent 决策链路
@app.post("/api/v1/chat")
async def chat(request: ChatRequest):
    with tracer.start_as_current_span("agent.chat") as span:
        span.set_attribute("user_id", request.user_id)
        span.set_attribute("model", request.model)

        # LLM 调用子链路
        with tracer.start_as_current_span("llm.call") as llm_span:
            llm_span.set_attribute("prompt_tokens", len(request.messages))
            response = await llm_client.chat(request.messages)
            llm_span.set_attribute("completion_tokens", response.usage.completion_tokens)

        # 工具调用子链路
        if response.tool_calls:
            for tc in response.tool_calls:
                with tracer.start_as_current_span(f"tool.{tc.name}") as tool_span:
                    result = await execute_tool(tc)
                    tool_span.set_attribute("tool_result", str(result)[:200])

        return response
```

### Jaeger 可视化

将 Span 数据发送到 Jaeger，可以在 Web UI 中看到 Agent 每个请求的完整调用瀑布图，直观定位瓶颈：

```python
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor

provider = TracerProvider()
processor = BatchSpanProcessor(OTLPSpanExporter(endpoint="http://jaeger:4317"))
provider.add_span_processor(processor)
trace.set_tracer_provider(provider)
```

## 指标收集

用 Prometheus 客户端收集 Agent 的关键指标：

```python
from prometheus_client import Counter, Histogram, Gauge
import time

llm_calls_total = Counter("llm_calls_total", "Total LLM API calls", ["model", "provider"])
llm_duration_seconds = Histogram("llm_duration_seconds", "LLM call latency", ["model"])
agent_errors_total = Counter("agent_errors_total", "Agent execution errors", ["error_type"])
active_sessions = Gauge("active_sessions", "Number of active Agent sessions")


@app.post("/api/v1/chat")
async def chat(request: ChatRequest):
    active_sessions.inc()
    start = time.monotonic()

    try:
        response = await agent.run(request.messages)
        llm_calls_total.labels(model="gpt-4", provider="openai").inc()
        llm_duration_seconds.labels(model="gpt-4").observe(time.monotonic() - start)
        return response
    except Exception as e:
        agent_errors_total.labels(error_type=type(e).__name__).inc()
        raise
    finally:
        active_sessions.dec()
```

## 生产环境日志架构

```
Agent Service → stdout (JSON 结构化日志)
              → OpenTelemetry Collector
                    → Jaeger (Tracing)
                    → Prometheus (Metrics)
                    → Elasticsearch (Logs)
```

本地开发时日志输出到控制台（人类可读格式），生产环境输出 JSON 格式由日志采集系统统一处理：

```python
import structlog
import logging

if config.ENV == "production":
    structlog.configure(
        processors=[
            structlog.stdlib.filter_by_level,
            structlog.stdlib.add_log_level,
            structlog.processors.TimeStamper(fmt="iso"),
            structlog.processors.JSONRenderer(),  # 生产环境用 JSON
        ],
    )
```

## 总结

可观测性是 Agent 系统生产化的基础设施。结构化日志让错误可搜索，分布式追踪让决策链路可视化，指标采集让趋势可量化。三者结合，才能在复杂 Agent 系统中快速定位问题，保障服务稳定性。
